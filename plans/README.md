## What each plan delivers

- **001 (contract):** a `Role` enum + per-wallet `Role`/`Active` storage; the
  fund-moving entrypoints (`create_shipment`, `accept`) auto-bind the caller's
  role and reject a conflicting role while the wallet has active services
  (`WrongRole`); terminal states (`deliver`, `refund_expired`) free the counts;
  `set_role`/`role_of`/`active_count` entrypoints; new errors `WrongRole = 23`,
  `RoleLocked = 24`; four new tests. **Requires a redeploy.**
- **002 (frontend cleanup):** removes the attacker role end-to-end (type, config,
  switcher, panel, `api.attack`, `/api/attack` route, server flow) and strips all
  "demo" framing (renames `app/demo`→`app/console`, `components/demo`→
  `components/console`, updates copy + the landing CTA), plus a localStorage guard
  so a stale `"attacker"` role can't rehydrate. Everything stays wired to the real
  deployed contract.
- **003 (frontend feature):** a first-connect modal with a role dropdown; the role
  switcher reads `active_count` from the contract and locks switching while the
  wallet has an active shipment, with a clear reason; reflects any on-chain-bound
  role. Selection is client-side (no tx); the contract (001) is the backstop.

## Design decisions worth knowing (so nobody re-litigates)

- **On-chain role binding covers merchant + carrier only** — the two roles that
  transact on the registry (DESIGN.md §3). Recipient and auditor never transact
  on-chain, so they are client-side roles; `role_of` returns null and
  `active_count` stays 0 for them (the switcher never locks them). This is
  intentional, not a gap.
- **Selection is instant / client-side; enforcement is on-chain at action time.**
  `create_shipment`/`accept` auto-bind and reject conflicts, so exclusivity holds
  regardless of the UI. Requiring a `set_role` transaction just to pick a role was
  considered and deferred (friction: a signed tx before you can use the app) —
  the `set_role` entrypoint exists (001) if the product later wants that; see
  003's maintenance notes.
- **No attacker role, by design.** Malicious actions are caught by the existing
  guards with real error codes (`BadProof`, `WrongState`, `NullifierSpent`,
  `TsBeforeAccept`, `require_auth`, and now `WrongRole`/`RoleLocked`). A scripted
  attacker role added nothing the protocol doesn't already enforce.

## Findings considered and rejected

- *Require an on-chain `set_role` tx at role-pick time* — deferred, not rejected:
  it adds friction (a signed tx to pick a role) for marginal benefit over
  auto-bind. The entrypoint ships in 001; wiring it is documented in 003 as the
  stronger alternative if wanted.
- *Bind recipient/auditor on-chain* — rejected: they never call a registry
  entrypoint, so there is nothing to enforce; binding them would cost gas for no
  guarantee.
